Design Decisions

Decision Reuse v4 formatInfo verbatim

Ambiguity: The annotation format [AS<n> <ASName> <prefix> | <country> <region> <city>] is identical between v4 and v6, and enrich.IPInfo is already dual-stack (#40).

Choice: Copied formatInfo unchanged into the v6 package rather than factoring a shared helper.

Rationale: The project convention is fully independent paired packages (ping/ping6, sonar/sonar6); a shared format helper would break that boundary for marginal dedup gain. The only v6-specific touch is example prefixes in the doc comment.

Deviations

Deviation Header reads "traceroute6 to", Summary responder column widened

Spec: AC says header traceroute6 to <host> (<v6 addr>), N hops max, <proto> probes.

Implemented: Header matches (with the v4 layout's trailing "N byte packets"); the Summary() responder column padding grew from %-15s (v4) to %-39s to fit a full IPv6 address.

Why: 15 columns truncate/misalign v6 addresses; 39 is the max textual width of an IPv6 address, keeping the stat columns aligned.

Tradeoffs

Tradeoff Inline ECMP responders vs one line per responder

Alternatives: (a) print all responders for a hop inline on the TTL line (v4 behavior); (b) emit a separate indented line per responder.

Chosen: (a), matching v4 writeHopLine exactly — continuation responders are space-aligned inline.

Why: Output parity with cmd/traceroute was the explicit goal (US choice 4A); diverging the layout for v6 would surprise users switching between the two tools.
