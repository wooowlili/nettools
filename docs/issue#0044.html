Issue #44 — 实现 Tracer 并发主循环 + finalize + enrich 接入 — 2026-06-30

Design Decisions

Decision Run-loop code landed with #43; #44 contributes test coverage

Ambiguity: #43 (probes) and #44 (run loop) share tracer.go; the loop, finalize, and enrich wiring were the connective tissue needed to make #43's probes runnable, so they were written together.

Choice: The concurrent Run/traceTarget/finalize + reverse-DNS/enrich hooks shipped in #43; #44 adds run_test.go proving the acceptance criteria (stats, dedup, reached/finalize, non-v6 placeholder) against synthetic results.

Rationale: Splitting the loop into a separate non-buildable commit would have left #43 unable to compile. Per-issue PRs stay green; #44's value is the synthetic-result test matrix the AC explicitly calls for.

Deviations

Deviation Live DNS/enrich paths exercised via unit hooks, not network calls

Spec: "reverse-DNS unified once over all hops" and "enrichment over deduped hop IPs once."

Implemented: The wiring is present and unit-tested for the dedup/finalize/stats logic with synthetic Results; the actual PTR and enrich.Resolve calls are not driven in tests (they hit the network).

Why: Mirrors the v4 traceroute test approach — keep unit tests hermetic; real resolution/enrichment is validated during CLI integration (#46).

Tradeoffs

Tradeoff Per-hop fixed slice + appended Probes under one mutex

Alternatives: (a) channel-collected results merged after the wait; (b) pre-sized per-hop slice with a shared probeMu guarding Probes append + addAddr.

Chosen: (b), matching v4 traceroute exactly.

Why: The hop index is fixed per goroutine, so only the per-hop Probes/Addrs mutation races; a single coarse mutex is simpler than a channel fan-in and the contention is trivial relative to the network wait.
